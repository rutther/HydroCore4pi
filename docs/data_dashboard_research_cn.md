# 数据仪表研究记录

日期：2026-07-05

## 结论先行

数据仪表不能设计成“实时通讯状态屏”。

当前系统的采集、存储、展示是解耦的：采集线程按计划读取传感器，成功后把数值写入 SQLite；前端只通过 API 读取采集计划视图、数据库序列摘要和时序数据。普通仪表页不能凭空知道“某个传感器此刻通讯正常”，只能根据数据库事实表达：

- 哪些参数在当前采集计划中；
- 哪些参数数据库里有数据；
- 每条参数序列最后一次成功落库是什么时候；
- 最近一段时间有没有数据缺口；
- 事件类参数是否最近发生过事件；
- 趋势图在所选时间范围内有哪些数据点。

因此页面文案和信息架构必须避免“通讯状态正常”“设备在线”这类过度承诺。更准确的说法是“最近成功采集”“最近落库”“当前计划包含”“暂无数据”“事件最近未发生”。

## 真实数据链路

```text
tasks/config_poll_plan.json
  -> /api/v1/meta/plan_view
  -> 前端参数清单、单位、名称、event_only、agg_mode

config_poller.py
  -> start_config_get 读取实际传感器
  -> 成功结果进入窗口累计
  -> 到 persist_sec 后写入 sensor_data
  -> 同步维护 sensor_series_summary

sensor_data / sensor_series_summary
  -> /api/v1/meta/series
  -> /api/v1/data/series
  -> 前端趋势图、最近值、样本数量
```

## 后端事实

### 1. `sensor_data` 只存事实数据

表字段是：

- `ts`
- `protocol`
- `address`
- `parameter`
- `value`
- `raw_hex`

它没有“通讯状态”“传感器在线状态”“报警解释文本”这些字段。

### 2. `sensor_series_summary` 只是序列摘要

字段是：

- `protocol`
- `address`
- `parameter`
- `first_ts`
- `last_ts`
- `n`

它能回答“这条序列有没有数据、最早/最新数据是什么时候、样本数多少”，不能回答设备此刻是否在线。

### 3. `plan_view` 才是参数定义入口

`/api/v1/meta/plan_view` 以采集计划为事实来源，并补充：

- `label`
- `unit`
- `round_to`
- `axis`
- `event_only`
- `agg_mode`

前端应优先使用这里的参数名称和单位，不应该自己根据字段名猜“水质/温度/腐蚀”之类分类。

### 4. `poller/status` 只代表采集线程

`/api/v1/poller/status` 当前只返回：

- `enabled`
- `running`

它说明采集线程是否启用/运行，不等价于“所有传感器通讯正常”。

## 当前运行数据验证

树莓派 `192.168.0.110` 当前返回：

- `plan_view`：3 个采集对象，地址 10、9、8，共 12 个参数；
- `meta/series`：12 条数据库序列；
- `poller/status`：`enabled=true`，`running=true`；
- 连续量最新落库时间约为 `2026-07-05 23:31:30`；
- `warning` 是事件类参数，最后一次落库时间明显早于连续量，这是事件存储策略导致的，不应被当成采集过期。

## 已发现的真实问题

### 1. 秒级分桶时间偏移

`raw` 查询返回的时间是正确的，例如：

```text
2026-07-05 23:00:00
2026-07-05 23:00:10
```

但 `bucket=10m&agg=last` 返回：

```text
2026-07-06 07:00:00
2026-07-06 07:10:00
```

这说明秒级分桶 SQL 把本地时间字符串当 UTC 转 epoch，再用 `localtime` 转回，导致时间整体偏移。趋势图如果使用分桶数据，会出现时间轴不可信。

### 2. 前端仍有字段名猜测逻辑

`ui/lab/js/dashboard.js` 中仍存在：

- `unitOf()`
- `parseKey()`

它会根据 `parameter` 名称猜单位和短标签。这只能作为最后兜底，不能作为主要显示逻辑。

### 3. 左侧默认显示 24h 变化不可靠

当前前端会批量请求 24h 数据并计算变化值/百分比。问题是：

- 事件类参数不适合显示 24h 连续变化；
- pH、电位、偏移量等参数不一定适合显示百分比变化；
- 基准值接近 0 时百分比会误导；
- 这会让现场人员把“展示算法结果”误认为设备诊断结论。

### 4. 事件类参数必须单独表达

`warning` 在计划里是：

```json
{
  "agg_mode": "last",
  "event_only": "nonzero_or_change",
  "label": "报警",
  "name": "warning"
}
```

它不是普通连续量。长时间没有新落库可能代表“无新事件”，不能直接解释成“采集异常”。

## 新设计边界

### 数据仪表应该表达

- 当前计划中有哪些采集参数；
- 每个参数的最新数据库值；
- 最新落库时间距离现在多久；
- 参数是连续量还是事件量；
- 所选参数的趋势；
- 数据缺口、无数据、事件发生等数据库可证明的状态；
- 必要时提供“来源详情”，显示协议、地址、字段名。

### 数据仪表不应该表达

- 某个传感器此刻通讯正常；
- 某个从站在线；
- 串口实时质量；
- 未经过数据库或专门诊断接口证明的设备健康结论；
- 固定“水质/温度/腐蚀”分类；
- 把协议、地址、字段名拆开给普通用户选择。

### 合理文案

推荐：

- `最近成功采集`
- `最新数据`
- `当前计划`
- `暂无数据`
- `无新事件`
- `数据缺口`
- `来源详情`

避免：

- `通讯正常`
- `设备在线`
- `实时连接`
- `数据新鲜窗口`
- `聚合`
- `协议数据字段`

## 下一步设计方向

先不画“工业通讯状态屏”，而是设计“数据库事实仪表”：

1. 顶部只展示系统可证明的摘要：采集线程运行、最新落库时间、计划参数数、无数据数、事件数。
2. 左侧为“当前计划参数”，每项来自 `plan_view`，再与 `meta/series` 合并显示数据库状态。
3. 参数状态只按数据库事实判断：有最新值、无数据、可能过期、事件无新记录、事件有记录。
4. 右侧为所选参数的当前值、趋势和数据质量，不显示未经证明的通讯结论。
5. 来源详情默认收起，里面才显示 `protocol/address/parameter/port`。
6. 修复分桶时间偏移后，再把趋势图作为可信视图。
