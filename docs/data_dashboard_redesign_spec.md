# HydroCore 3.1 数据仪表改造规格

日期：2026-07-05

关联分析：

- `docs/system_analysis_data_dashboard.md`

## 1. 改造目标

数据仪表不是普通图表页，而是现场人员判断采集系统是否可信的主界面。

它的第一目标：

> 让用户在 10 寸机置屏幕上，第一眼知道采集是否正常、数据是否新鲜、哪些参数可用、哪些参数异常。

它的第二目标：

> 让用户在确认数据可信后，再进入趋势图、回放、统计、导出等分析动作。

它不承担：

- 编辑协议文件。
- 编辑采集计划。
- 配置任务计划。
- 配置输出动作。
- 执行真实设备控制。

## 2. 当前证据

当前树莓派接口状态：

- `/api/v1/poller/status`: `enabled=true`, `running=true`
- `/api/v1/meta/plan_view`: 3 个采集对象
- `/api/v1/meta/series`: 12 条序列，`source=summary`

当前采集对象：

| 地址 | 协议 | 参数 |
| --- | --- | --- |
| 10 | `lanchang_ec` | 电导率、电阻率、温度、TDS、盐度 |
| 9 | `lanchang_ph` | pH、温度、电流输出、报警 |
| 8 | `lanchang_cr_huchen` | 腐蚀率、电位、偏移量 |

当前已发现的页面问题：

- `报警` 是事件量，`event_only=nonzero_or_change`，不应该按普通连续量显示 24h 变化。
- `报警` 最新落库时间不等于采集停止，它只表示最近一次事件写入。
- `电位` 出现类似 `+70.785 (+-92.39%)` 的格式问题。
- 前端仍有 `unitOf()`、`parseKey()` 这类按字段名猜单位/标签的兜底逻辑。
- 1024x600 近似 10 寸场景下，控制区高度过大，趋势图空间被压缩。
- 左侧列表像行情 ticker，不像工业 HMI 的参数状态清单。

## 3. 用户心智

现场用户进入数据仪表时，不应该先思考“我要选哪条曲线”。

正确心智是：

1. 这台设备采集有没有在跑。
2. 最新数据是不是刚刚采到的。
3. 哪个传感器或参数不正常。
4. 点开某个参数看趋势。
5. 必要时导出或回放历史。

因此页面层级应从“图表优先”调整为“状态优先”。

## 4. 参数身份模型

前端显示和任务计划引用参数时，统一使用完整参数身份。

完整参数身份：

```json
{
  "key": "lanchang_ph:9:measurement",
  "display_name": "pH @ 地址9",
  "label": "pH",
  "unit": "pH",
  "protocol": "lanchang_ph",
  "address": 9,
  "port": "/dev/ttyACM0",
  "parameter": "measurement",
  "value_kind": "continuous",
  "event_only": null,
  "agg_mode": null,
  "round_to": 3
}
```

用户只看到：

- `pH @ 地址9`
- `温度 @ 地址9`
- `电导率 @ 地址10`

系统内部继续保留：

- protocol
- address
- parameter
- port
- event_only
- agg_mode

禁止在普通界面让用户分别选择协议、地址、字段。

## 5. 参数类型

数据仪表必须区分三类参数。

### 5.1 连续测量值

适合显示：

- 当前值
- 最近更新时间
- 24h 变化
- 趋势图
- 平均/最小/最大/最后

例子：

- pH
- 电导率
- 电阻率
- 温度
- TDS
- 盐度
- 腐蚀率
- 电位

但注意：

- 百分比变化不是所有连续量都适合显示。
- 跨 0 或初始值接近 0 时，应隐藏百分比。

### 5.2 事件/状态量

识别依据：

- `event_only` 存在且非空。
- 或协议/计划明确 `value_kind=status/event`。

适合显示：

- 当前状态
- 最近事件时间
- 最近 24h 事件次数
- 是否有未恢复报警

不适合默认显示：

- 24h 连续变化百分比
- 连续折线趋势

当前例子：

- `warning / 报警`

### 5.3 配置/校准量

这类参数来自协议 JSON 中的可写配置项，通常不在采集计划里。

普通数据仪表不显示它们。

它们属于：

- 硬件配置
- 设备定义校验
- 高级诊断

## 6. 数据健康状态模型

每条参数序列需要计算一个状态。

建议状态：

| 状态 | 条件 | 用户显示 |
| --- | --- | --- |
| `fresh` | poller 运行，连续量最新时间在有效期内 | 新鲜 |
| `stale` | poller 运行，但连续量超出有效期 | 过期 |
| `no_data` | 计划中存在，但数据库没有数据 | 无数据 |
| `event_idle` | 事件量长时间无事件，但采集正常 | 无新事件 |
| `event_active` | 事件量最近有非零或未恢复值 | 有报警 |
| `poller_stopped` | poller 停止 | 采集停止 |
| `unknown` | 无法判断 | 未知 |

连续量默认有效期建议：

- `max(3 * sampling_sec, 60 秒)`
- 如果后端拿不到采样周期，则默认 180 秒。

事件量不能用连续量规则判断过期。

## 7. 后端 API 建议

### 7.1 保留现有接口

继续保留：

- `/api/v1/meta/plan_view`
- `/api/v1/meta/series`
- `/api/v1/data/series`
- `/api/v1/data/export.csv`

### 7.2 新增或扩展一个面向仪表的状态接口

建议新增：

`GET /api/v1/meta/dashboard_state`

返回：

```json
{
  "ok": true,
  "server_ts": "2026-07-05 22:07:10",
  "poller": {
    "enabled": true,
    "running": true
  },
  "devices": [
    {
      "protocol": "lanchang_ph",
      "address": 9,
      "port": "/dev/ttyACM0",
      "display_name": "PH 传感器 @ 地址9",
      "series_count": 4,
      "fresh_count": 3,
      "stale_count": 0,
      "event_count": 1,
      "last_ts": "2026-07-05 22:07:10",
      "status": "fresh"
    }
  ],
  "series": [
    {
      "key": "lanchang_ph:9:measurement",
      "display_name": "pH @ 地址9",
      "label": "pH",
      "unit": "pH",
      "protocol": "lanchang_ph",
      "address": 9,
      "port": "/dev/ttyACM0",
      "parameter": "measurement",
      "value_kind": "continuous",
      "last_ts": "2026-07-05 22:07:10",
      "age_sec": 6,
      "status": "fresh",
      "n": 127162,
      "latest_value": 8.51,
      "show_delta": true,
      "show_percent_delta": false
    }
  ]
}
```

这样前端不用每次自己拼 `plan_view + meta/series + data/series` 才知道状态。

### 7.3 第一阶段如果不新增接口

如果先不新增接口，前端可以临时合并：

- `plan_view` 提供参数定义。
- `meta/series` 提供 `first_ts / last_ts / n / server_ts`。
- `/data/series?bucket=10m&agg=last` 提供当前值和 24h 变化。

但这只能作为过渡。长期应该把状态判断放回后端，减少前端复杂度和树莓派浏览器压力。

## 8. 10 寸机置屏信息架构

目标尺寸：

- 10 寸主屏。
- 以 1024x600 到 1280x800 范围优先。
- 手机仅作为应急查看。

### 8.1 页面结构

推荐结构：

```text
┌──────────────────────────────────────────────┐
│ 顶部状态条：采集中 / 最新6秒前 / 12项 / 0异常 │
├───────────────┬──────────────────────────────┤
│ 参数状态清单   │ 趋势与详情                    │
│               │                              │
│ 传感器分组/过滤│ 当前参数摘要                   │
│ 参数卡片       │ 最近/回放 + 间隔 + 统计        │
│ 状态徽标       │ 图表                           │
│               │ 数据缺口/导出                   │
└───────────────┴──────────────────────────────┘
```

### 8.2 顶部状态条

必须显示：

- 采集运行状态。
- 最新连续量更新时间。
- 参数总数。
- 异常/过期/无数据数量。

示例：

`采集中 · 最新 6 秒前 · 11 个连续量正常 · 1 个事件量无新事件`

不要只显示：

`采集中 · 6秒前`

因为这无法解释事件量和异常数量。

### 8.3 左侧参数状态清单

每个参数项建议显示：

- 参数名：`pH @9`
- 当前值：`8.51 pH`
- 状态：`新鲜 / 过期 / 无数据 / 无新事件 / 有报警`
- 最近更新：`6秒前`
- 24h 变化：只对适合参数显示

左侧不要叫“指标”也可以考虑改为：

- `参数`
- `采集参数`
- `监测参数`

更推荐 `监测参数`。

### 8.4 右侧趋势与详情

右侧默认显示当前选中参数：

- 参数完整身份。
- 来源：协议、地址、端口，放在高级/详情里。
- 当前值和更新时间。
- 最近趋势。
- 数据缺口提醒。

控制区压缩为两行：

第一行：

- 最近：1H / 3H / 12H / 24H / 7D
- 回放

第二行：

- 间隔
- 统计
- 刷新
- 导出

避免控制区超过 110px。

## 9. 手机应急视图

手机窄屏不承载完整分析。

优先显示：

1. 采集状态。
2. 异常/过期参数。
3. 参数当前值。
4. 单参数趋势。

手机可以保留：

- 刷新
- 最近 1H/24H
- 查看详情

手机不优先支持：

- 多参数复杂对比
- 大范围回放编辑
- 高级统计配置

## 10. 前端性能策略

必须保留自动刷新，但避免请求风暴。

建议：

- 状态接口每 10 秒刷新一次。
- 当前值使用状态接口返回，避免每个参数单独请求。
- 趋势图只请求选中的参数。
- 最近 24h 左侧变化不要每次全量重算所有参数。
- 长时间窗口降低刷新频率。
- 页面不可见时暂停图表刷新，恢复可见时补一次。
- 刷新请求必须互斥，上一轮未结束时合并下一轮请求。

当前 `refreshAllPromise / refreshAllPending` 方向是对的，应保留。

## 11. 图表策略

图表应当支持：

- 单参数趋势。
- 少量多参数对比。
- 数据断点显示。
- 回放。
- 导出。

图表不应承担：

- 参数健康判断。
- 事件量主显示。
- 系统运行状态表达。

事件量如果需要画图，应该进入事件详情视图，而不是普通连续折线。

## 12. 第一阶段实施范围

第一阶段只做数据仪表，不改硬件配置、动作配置、任务计划。

建议顺序：

1. 修正前端参数元数据使用：
   - `event_only` 非空即识别为事件量。
   - `unitOf()` 只做兜底，主显示来自 `plan_view`。
   - 参数对象保留 `protocol/address/parameter/port/event_only/agg_mode`。

2. 增加序列状态计算：
   - 连续量：新鲜/过期/无数据。
   - 事件量：无新事件/有事件。
   - 顶部显示汇总状态。

3. 修正左侧参数卡：
   - 事件量不显示 24h 变化。
   - 不适合百分比的参数隐藏百分比。
   - 修复 `+-` 符号。

4. 压缩 10 寸屏控制区：
   - 控制区两行内。
   - 状态条独立。
   - 图表高度稳定。

5. 再考虑是否新增 `/meta/dashboard_state`：
   - 如果前端状态计算变复杂，就把判断前移到后端。

## 13. 验收标准

### 13.1 功能验收

- 页面能显示 poller 是否运行。
- 页面能显示最新连续量数据距当前多久。
- 页面能区分连续量和事件量。
- `warning / 报警` 不再按普通连续量展示 24h 百分比变化。
- 参数名称和单位来自采集计划/协议链路，不由前端硬编码生成。
- 点击参数可以看到趋势。
- 自动刷新仍然存在。
- 请求不会重叠堆积。

### 13.2 10 寸屏验收

- 1024x600 下不出现横向溢出。
- 1024x600 下顶部控制区不挤掉图表主体。
- 左侧列表、右侧图表同时可见。
- 状态条能完整读懂。
- 常用操作不需要多层滚动。

### 13.3 语义验收

- 普通界面不出现后端术语。
- 不出现“数据新鲜窗口”“聚合”等术语。
- 统计方式可以叫“统计”，不要叫“聚合”。
- 采集冲突应表达为“采集中，不能修改设备配置”，不要表达为锁或线程错误。

## 14. 当前判断

数据仪表不需要推倒重写。

它当前的底层能力已经基本够用：

- 元数据有 `plan_view`。
- 序列范围有 `meta/series`。
- 趋势有 `data/series`。
- 自动刷新已有互斥保护。
- ECharts 可以继续使用。

真正要改的是产品层：

1. 从图表页改成状态优先的 HMI。
2. 从字段猜测改成配置链路驱动。
3. 从行情 ticker 改成参数健康清单。
4. 从漂亮曲线改成可信数据。
