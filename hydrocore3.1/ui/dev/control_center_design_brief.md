# HydroCore 3.1 控制中心设计 brief

## 1. 范围

这次只重做两个现有大页的产品层和交互层：

- `动作配置`
- `任务计划`

不重做整套系统，不改数据仪表、硬件配置、系统设置的主结构。

## 2. 已确认约束

- 保留现有黑底工业 HMI 外壳，不做营销站风格，不做浅色 SaaS 风格。
- 保留现有总壳体规则：
  - 横屏主工作台：`1280 x 800`
  - 竖屏巡检/修改：`800 x 1200`
- 保留多语言能力，前台文案必须优先中文可理解性。
- 保留后端现有兼容层：
  - `action_unit`
  - `task`
  - `rule`
  - `schedule`
- 前台不直接把这些后端术语甩给用户。

## 3. 产品判断

### 3.1 动作配置

用户真正要做的是三件事：

1. 先确认这套硬件的可用 GPIO 规则
2. 再定义“哪个 GPIO 对应哪个设备”
3. 最后定义“这个设备一次要怎么动作”

所以前台结构固定为：

1. `硬件预设`
2. `输出设备`
3. `动作模板`

其中：

- `输出设备` 回答“我控制的是谁”
- `动作模板` 回答“它一次怎么动”

`动作单元` 改名为 `动作模板`，因为用户理解的是“模板/动作方案”，不是“单元”。

### 3.2 任务计划

用户真正要做的是：

- 决定什么时候执行
- 决定执行哪个动作模板
- 决定哪些限制必须生效

所以前台不再拆成：

- 任务
- 触发规则
- 定时计划

而是合并成一个统一对象：`计划`

每条计划只有三块信息：

1. `执行内容`
2. `触发方式`
3. `安全限制`

后端仍可保留 task/rule/schedule 兼容层，但前台只暴露“计划”。

## 4. 前台术语替换

以下术语前台禁用或弱化：

- `dry-run`
- `thread`
- `automation`
- `cooldown`
- `aggregation`
- `fresh data window`

统一改成：

- `仅模拟`
- `调度服务`
- `自动执行`
- `两次执行至少间隔`
- `统计方式`
- `数据有效期`

## 5. 横屏布局原则

### 动作配置

横屏使用三段式结构：

1. 左侧列表：对象列表
2. 中间主编辑区：当前对象表单
3. 右侧影响区：接线摘要、使用关系、执行预览

右侧影响区必须存在，因为动作配置最容易发生“改了一个设备却不知道哪些动作和计划会受影响”。

### 任务计划

横屏使用三栏工作台：

1. 左栏：计划列表
2. 中栏：计划编辑器
3. 右栏：运行与安全侧栏

右栏固定承担三件事：

- 当前执行预览
- 触发/限制摘要
- 最近一次执行结果

这样用户不用滚到页面底部找“刚才到底会不会跑”。

## 6. 竖屏布局原则

竖屏不是把横屏三栏压成一长列，而是改成分步卡片：

1. 先选对象
2. 再编辑当前步骤
3. 最后看执行与状态

任务计划竖屏优先顺序：

1. 名称和启停
2. 触发方式
3. 执行动作
4. 安全限制
5. 执行按钮与最近结果

## 7. 用户流程

### 7.1 新增输出设备

1. 打开 `动作配置`
2. 进入 `输出设备`
3. 点 `新建设备`
4. 选择类型、GPIO、默认状态
5. 保存
6. 在右侧看到“暂未被动作模板使用”

### 7.2 新增动作模板

1. 打开 `动作配置`
2. 进入 `动作模板`
3. 点 `新建模板`
4. 选目标设备
5. 选动作模式
6. 填持续时间或循环参数
7. 保存
8. 在右侧看到执行摘要

### 7.3 新增计划

1. 打开 `任务计划`
2. 点 `新建时间` 或 `新建条件`
3. 填计划名称
4. 选动作模板
5. 设触发方式
6. 设安全限制
7. 保存
8. 可选 `检查` 或 `试跑`

### 7.4 取消新建

必须始终存在明确的 `取消新建`

禁止依赖：

- `刷新`
- 回到别的页再回来
- 隐式点左侧列表

## 8. 这轮设计稿的交付物

设计稿页应至少覆盖：

1. 横屏 `动作配置 / 输出设备`
2. 横屏 `动作配置 / 动作模板`
3. 横屏 `任务计划 / 计划列表 + 编辑器 + 运行侧栏`
4. 竖屏 `任务计划 / 单计划编辑`

## 9. 实现核对标准

实现时逐项核对：

1. 前台是否仍暴露程序员术语
2. 是否仍需要靠 `刷新` 才能退出新建
3. 是否仍把时间计划和条件计划拆成割裂大页
4. 是否仍把说明文字大段贴在页面中央
5. 横屏下是否存在明确主次层级
6. 竖屏下是否只是简单换行
7. 用户能否在不理解后端结构的情况下完成新增计划
