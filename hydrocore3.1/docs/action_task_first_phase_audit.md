# HydroCore 3.1 动作配置与任务计划第一阶段审计方案

日期：2026-06-17

## 0. 产品简报

本阶段只处理两个现有路由页：

- 动作配置：定义设备“能怎么动”。
- 任务管理：改造成用户能理解的“任务计划”，定义“什么时候、因为什么去动”。

不重做整个系统，不重做数据仪表、硬件配置、系统设置。现有 WebUI、后端 API、数据文件结构是约束，不是另起炉灶的理由。

## 1. 当前结论

动作配置页的方向基本是对的，但把运行状态、自动控制、任务、规则、计划混进来了，导致边界变脏。

任务管理页目前是后端对象管理器，不是给现场用户使用的控制计划页面。它把 task、rule、schedule、thread、dry-run、fresh-data window、aggregation 这些程序员概念直接暴露给用户，用户很难知道自己到底是在配置“什么时候执行什么动作”，还是在调试后端结构。

第一阶段不应继续扩展复杂能力，而应先把概念收敛：

- 动作单元：一个可复用的设备动作，例如“加药泵 A 运行 3 秒”。
- 组合动作：多个动作单元按顺序执行，例如“打开排水阀，等待 20 秒，关闭排水阀”。
- 任务计划：一个用户可理解的自动执行安排，包含执行内容和触发条件。
- 触发条件：按时间触发，或按水质/传感器条件触发。
- 运行记录：解释已经发生什么、为什么发生、为什么没发生。

后端可以继续保存 task/rule/schedule，但普通用户界面不要把它们分成三个一级对象。

## 2. 前端现状审计

### 2.1 `ui/js/pages/action_config/template.js`

当前动作配置有四个子页：设备预设、总览、输出设备、动作单元。

问题：

- “总览”里出现自动控制、线程启停、dry-run、真实输出、任务/规则/计划数量，这些不属于动作配置的核心任务。
- 设备预设页把 JSON 预览放得过重。现场用户首先需要确认“当前启用的输出能力”，不是阅读 JSON。
- 动作单元编辑页把毫秒、relay_pulse、relay_pattern、PWM 等实现术语直接推给用户。

第一阶段处理：

- 保留设备预设、输出设备、动作单元。
- 删除或降级“总览”中的自动控制/任务/规则/计划信息。
- JSON 预览折叠到“高级信息”。
- 动作单元表单改成人话：运行多久、打开/关闭、按周期运行、只演练不实际输出。

### 2.2 `ui/js/pages/action_config/runtime-panel.js`

这个文件承担了过多职责：动作配置、任务、规则、计划、日志、自动线程都在里面有逻辑残留。

问题：

- 即使模板里没有显示任务/规则/计划子页，逻辑依旧存在，造成维护者误判。
- 动作配置和自动调度耦合，后续改 UI 容易牵连后端行为。

第一阶段处理：

- 只保留设备预设、输出设备、动作单元需要的状态和事件。
- 与任务、规则、计划、日志相关的函数迁出或停止在动作配置中使用。
- 不改后端行为，只清理前端入口和用户心智。

### 2.3 `ui/js/pages/tasks.js`

当前任务管理页包含：summary、automation panel、tasks/rules/schedules/logs 四个 tab。

问题：

- “任务 / 触发规则 / 定时计划”是数据库/后端对象拆法，不是用户任务拆法。
- task 编辑器与动作单元功能高度重叠，尤其 single_action task 对用户没有额外意义。
- rule 编辑器暴露 metric_key、signal_protocol、signal_address、aggregation、fresh data 等底层概念。
- schedule 编辑器只支持 once/daily/interval，却容易让用户误以为能处理复杂营业时段。
- automation panel 暴露“启动线程/停止线程”，这是后台服务状态，不是业务操作。

第一阶段处理：

- 页面标题从“任务管理”调整为“任务计划”。
- 一级结构改为“计划列表 / 运行记录 / 高级诊断”。
- 普通编辑器统一为“新建计划”，先选触发方式：
  - 按时间执行：一次、每天固定时间、每隔一段时间。
  - 按水质条件执行：选择指标、条件、持续多久、执行动作。
- single_action task 在普通界面隐藏，由前端适配层自动创建或复用。
- sequence task 以“组合动作”名义保留，不作为第一屏核心。
- thread/dry-run/fresh/aggregation/cooldown 改成用户语言或放入高级区。

### 2.4 `ui/i18n/zh-CN.json` 与 `ui/i18n/en-US.json`

当前中英文都存在明显的后端术语污染。

必须替换的中文：

- dry-run -> 演练模式 / 只记录，不控制设备
- 启动线程 -> 重启调度服务（仅高级诊断）
- 停止线程 -> 暂停调度服务（仅高级诊断，默认不放普通操作区）
- 数据新鲜窗口 -> 只使用最近 N 秒的数据
- 冷却时间 -> 两次执行至少间隔
- 时间聚合 -> 判断方式
- 允许真实输出 -> 允许控制继电器/PWM
- 任务/规则/计划 -> 普通 UI 统一为任务计划，底层对象只在高级信息出现

英文也要同步改，不允许中文修了但英文仍是 backend copy。

### 2.5 响应式与尺寸

当前尺寸规则主要来自：

- `tokens.css`：最小宽度 580，最大宽度 1280；横屏 shell 高度 800，窄屏 shell 高度 1200。
- `layout.css`：小于 800px 时切到竖屏高度。
- `app.css` 与 `action_config.css`：主要断点是 799、980、1100、1200px。

风险：

- 当前没有看到按长宽比触发的专门逻辑。如果你原先设想过“横屏/竖屏按比例判断”，现在代码里没有稳定体现。
- 当前大量内部滚动区域会让横屏 1280x800 下的操作区割裂。

第一阶段规则：

- 横屏优先保证左列表 + 右编辑器可读，不再出现多个内嵌滚动同时争抢。
- 竖屏改为上下布局：列表在上，编辑器在下。
- 不用药丸形大按钮；卡片和表单控件圆角控制在 8px 左右。

## 3. 后端现状审计

### 3.1 `backend/api/action_api.py`

已提供完整 CRUD 和执行接口：

- `/summary`
- `/automation/status`
- `/automation/start`
- `/automation/stop`
- `/automation/config`
- `/actuators`
- `/units`
- `/units/<id>/execute`
- `/tasks`
- `/tasks/<id>/execute`
- `/rules`
- `/rules/<id>/evaluate`
- `/schedules`
- `/schedules/<id>/trigger`
- `/logs`

结论：第一阶段不需要重写后端 API。前端应该做产品适配，而不是把后端对象原样展示。

### 3.2 `backend/services/action_store.py`

后端实际数据模型是合理的底层模型：

- actuator：继电器或 PWM 输出设备。
- action_unit：一个动作单元。
- action_task：single_action 或 sequence。
- action_rule：传感器条件触发。
- action_schedule：时间触发。

问题不在模型本身，而在模型被直接暴露给用户。

### 3.3 `backend/services/action_scheduler.py`

调度器当前支持：

- 自动控制启用/禁用。
- 演练模式。
- 硬件总闸。
- once/daily/interval 时间计划。
- last/avg/min/max 条件判断。
- 冷却与每小时最大执行次数。

当前已补齐：

- 支持每个时间计划/水质条件计划配置生效日。
- 支持每天生效开始时间与结束时间。
- 支持结束时间早于开始时间的跨午夜窗口，例如 21:00-06:00。

仍有限制：

- 不支持复杂节假日/例外日期。
- 不支持跨计划复用的统一运行日历模板。

UI 只能展示后端已经支持的生效日/生效时段，不展示节假日或复杂日历假功能。

## 4. 用户概念与后端概念映射

| 用户看到 | 用户理解 | 后端对象 |
| --- | --- | --- |
| 输出设备 | 接在哪个 GPIO 上的泵、阀、PWM | actuator |
| 动作单元 | 某个设备怎么运行一次 | action_unit |
| 组合动作 | 多个动作按顺序执行 | action_task(sequence) |
| 任务计划 | 自动执行安排 | action_rule 或 action_schedule，加上 action_task |
| 按时间执行 | 到点或按间隔执行 | action_schedule |
| 按水质条件执行 | 指标满足条件后执行 | action_rule |
| 演练模式 | 只记录，不实际控制设备 | dry_run |
| 允许控制设备 | 真实输出总许可 | hardware_armed + allow_real_output |
| 两次执行至少间隔 | 防止重复触发 | cooldown_sec |
| 运行记录 | 执行/跳过/失败原因 | action logs |

## 5. 第一阶段信息架构

### 5.1 动作配置

页面目标：把“硬件输出能力”和“可复用动作”配置清楚。

子页：

- 设备预设
- 输出设备
- 动作单元

不在此页展示：

- 自动控制开关
- 线程启停
- 任务/规则/计划数量
- 执行日志主入口

### 5.2 任务计划

页面目标：让用户知道自动控制是否会运行、什么时候运行、为什么运行、运行了什么。

子页：

- 计划列表
- 运行记录
- 高级诊断

计划编辑器结构：

1. 计划名称
2. 执行内容：动作单元或组合动作
3. 触发方式：按时间 / 按水质条件
4. 生效时段：周几生效、每天几点到几点
5. 安全设置：演练模式、两次执行至少间隔、启用/停用
6. 说明

高级诊断只给维护者使用：

- 调度服务状态
- 最近轮询时间
- 后端 task/rule/schedule ID
- 原始 JSON

## 7. 验收标准

普通用户界面必须满足：

- 看不到 dry-run、thread、task/rule/schedule、aggregation 这些后端词。
- 动作配置页只回答“有哪些输出设备、每个设备怎么动作”。
- 任务计划页只回答“什么时候/因为什么执行什么动作”。
- 单动作计划不会要求用户先理解 task。
- 按时间计划可以覆盖当前后端支持的 once/daily/interval。
- 按条件计划可以覆盖当前后端支持的指标、比较、持续时间。
- 时间计划和水质条件计划都支持周几/时段生效。
- 跨午夜时段可用于夜间运行或夜间停机策略。
- 运行记录能看到执行成功、失败、跳过及原因。
- 中英文文案同步，不出现一半中文一半英文。
- 横屏 1280x800 不出现明显错位、遮挡、多层内滚动混乱。
- 窄屏按上下布局可操作。

## 8. 暂缓能力

以下能力重要，但不放入第一阶段：

- 节假日/例外日期。
- 可复用的运行日历模板。
- 多计划冲突仲裁策略。
- 多动作并发互锁。
- 复杂 PID 或闭环控制。

原因：当前先把单个计划的可用性做顺，再设计跨计划、跨日期的统一调度模型。
