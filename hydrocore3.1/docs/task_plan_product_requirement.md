# HydroCore 3.1 任务计划模块产品需求

日期：2026-06-21

## 1. 模块边界

本文只定义“任务计划”大功能模块。

不在本轮范围内：

- 数据仪表盘
- 硬件配置
- 动作配置
- 系统设置
- 整站视觉重做

任务计划只回答一个问题：

> 现场边缘控制器在什么时间、什么水质条件、什么安全边界下，自动执行哪个已经定义好的动作。

动作怎么运行，由“动作配置”负责。任务计划不重新定义 GPIO、继电器、PWM、动作时长和动作细节。

## 2. 产品角色

HydroCore 是一个循环水处理场景的工业物联网边缘端设备。它可以接入局域网，也必须可以在无公网、无云端、甚至临时断网时单体运行。

任务计划模块不是 SaaS 自动化平台，也不是程序员规则编辑器。它更接近控制柜上的本地自动运行策略面板。

设备可能安装在：

- 冷却塔机房
- 泵房
- 循环水设备柜
- 水处理控制箱
- 商业楼宇设备间
- 小型工厂或实验循环水现场

## 3. 购买者与使用者

### 3.1 购买者

- 水处理工程商或系统集成商：需要一套低成本、可交付、可本地运行的自动控制设备。
- 物业或工厂运维方：购买的是少人工巡检、少水质事故、少误操作、少设备损坏。
- 小型循环水系统负责人：希望用较低成本实现基础监测、加药和排水控制。

### 3.2 使用者

- 现场调试人员：接线后确认传感器、继电器、泵阀是否工作。
- 运维人员：查看自动计划是否开启、最近有没有执行、为什么没执行。
- 水处理技术人员：调整 pH、电导率等阈值，设置加药或排水策略。
- 远程维护者：在局域网内或临时接入后检查配置和记录。

## 4. 核心痛点

任务计划模块要解决的核心痛点不是“自由编程”，而是“可预期、可解释、可离线运行”。

用户真正担心的是：

- 不知道系统什么时候会自动控制泵阀。
- 不知道某条计划为什么执行或为什么没执行。
- 担心自动控制在不该动的时候动了。
- 现场断网后系统不能继续独立运行。
- 需要快速暂停自动计划，但保留人工点动能力。
- 不想理解 task、rule、schedule、thread、dry-run、aggregation、fresh window 等后端词。

## 5. 产品原则

1. 前端只讲用户语言，不展示后端对象拆法。
2. 所有自动行为必须能解释。
3. 自动计划和人工操作分开，不互相污染。
4. 普通路径简单，高级能力折叠。
5. 树莓派前端保持轻量，复杂判断放在后端。
6. 本地配置、本地调度、本地日志必须离线可用。
7. 高风险输出必须有清晰状态和确认路径。

## 6. 用户概念模型

任务计划统一使用“计划”这个用户概念。

计划由五部分组成：

1. 触发方式：什么时候尝试执行。
2. 生效时间：什么时候允许这条计划生效。
3. 执行动作：调用哪个动作配置。
4. 保护限制：什么情况下不能执行，或者最多执行多少。
5. 执行解释：最近一次为什么执行、跳过、拦截或失败。

推荐用户表达：

- 每天 08:00 排水 20 秒。
- 每 30 分钟检查一次，满足条件才加药。
- pH 高于 6.8 并持续 10 分钟后，加药泵 A 运行 3 秒。
- 周一到周六 06:00-20:00 允许自动加药。
- 夜间禁止自动加药。
- 维护中，暂停全部自动计划。

## 7. 任务计划类型

### 7.1 按时间执行

用于固定时间或固定间隔的自动动作。

普通模式需要支持：

- 执行一次
- 每天某个时间执行
- 每隔一段时间执行
- 每周选择星期几
- 每天选择允许时段

高级模式可以支持：

- 高级时间表达式
- 下次执行时间预览
- 表达式错误解释

普通用户不应直接面对 crontab 字符串，但系统需要保留接近 crontab 的表达能力。

### 7.2 按水质执行

用于传感器指标达到条件后执行动作。

普通模式需要支持：

- 选择指标，例如 pH、电导率、温度等。
- 选择比较方式，例如高于、低于。
- 输入阈值。
- 输入持续时间。
- 选择执行动作。
- 设置生效星期和生效时段。

高级模式可以折叠：

- 使用最新值、平均值、最大值、最小值。
- 数据有效时间。
- 每小时最多执行次数。
- 两次执行最少间隔。

这些概念在普通界面不应使用“时间聚合”“数据新鲜窗口”“冷却时间”这类术语。

### 7.3 保护限制

保护限制不是普通执行任务。它的作用是拦截某些自动动作。

典型例子：

- 夜间禁止自动加药。
- 维护中禁止全部自动输出。
- 排水阀打开时禁止加药。
- 传感器无数据时禁止按水质自动加药。
- 设备未授权时禁止自动控制继电器和 PWM。

保护限制应该在任务计划模块里可见，因为它直接影响计划是否执行。但它在 UI 上要明确标为“限制”，不能和“执行动作”的计划混在一起。

## 8. 优先级与冲突规则

可以有优先级，但不能只靠“上方覆盖下方”这种隐式逻辑。

推荐执行顺序：

1. 全局急停或维护状态。
2. 设备输出授权状态。
3. 保护限制。
4. 计划自身是否启用。
5. 计划的生效星期和时段。
6. 触发条件是否成立。
7. 频率限制。
8. 同一设备的互斥判断。
9. 执行动作。

计划列表可以按优先级排序，但每条计划必须显示类型：

- 执行计划
- 保护限制

当计划被拦截时，必须显示具体原因。例如：

> pH 偏高加药 A：条件已满足，但当前处于“夜间禁止自动加药”，所以未执行。

这比单纯显示“跳过”“失败”“被阻止”更重要。

## 9. 自动、手动、检查的边界

任务计划模块里必须把三件事分清楚。

### 9.1 检查配置

不控制真实设备。

用于检查：

- 计划参数是否合法。
- 动作是否存在。
- 时间设置是否可计算。
- 条件字段是否能读到数据。
- 下一次可能什么时候执行。

按钮文案：检查配置。

### 9.2 执行一次

真实控制设备。

用于现场确认这条计划调用的动作能真实运行。点击前需要清楚告诉用户会控制哪个设备、运行多久。

按钮文案：执行一次。

### 9.3 自动运行

由后台调度服务按计划执行。

用户需要能看到：

- 自动计划是否启用。
- 是否允许自动控制设备。
- 最近一次巡检时间。
- 最近一次执行或跳过原因。

不要在普通 UI 中出现“启动线程”“停止线程”。

## 10. 页面信息架构

任务计划页面建议分为三个子视图：

1. 计划
2. 运行记录
3. 高级诊断

### 10.1 计划

主工作区，默认打开。

横屏结构：

- 左侧：计划列表。
- 中间：当前计划编辑器。
- 右侧：当前计划判断结果、下次执行预览、最近记录。

竖屏结构：

- 上方：计划列表和状态筛选。
- 下方：计划详情。
- 详情内使用分段折叠，避免长表单失控。

计划列表每项显示：

- 计划名称
- 类型：时间执行、水质执行、保护限制
- 状态：运行中、已暂停、条件未满足、被限制、无数据
- 下次执行或最近结果
- 最近一次原因

### 10.2 运行记录

回答“刚才发生了什么”。

每条记录显示：

- 时间
- 计划名称
- 来源：自动、手动、检查配置
- 执行动作
- 结果：成功、跳过、拦截、失败
- 原因

### 10.3 高级诊断

给维护者和开发者使用，默认不打扰普通用户。

可以包含：

- 调度服务状态
- 后端对象 ID
- 原始 JSON
- 最近错误
- 巡检间隔
- 数据有效时间

## 11. 关键操作流程

### 11.1 新建时间计划

1. 点击新建计划。
2. 选择按时间执行。
3. 填计划名称。
4. 选择执行动作。
5. 选择执行方式：一次、每天、每隔一段时间。
6. 设置生效星期和时段。
7. 查看下次执行预览。
8. 保存。
9. 可选执行一次，真实验证设备动作。

### 11.2 新建水质计划

1. 点击新建计划。
2. 选择按水质执行。
3. 选择监测指标。
4. 设置高于或低于某个值。
5. 设置持续多久才成立。
6. 选择执行动作。
7. 设置生效星期和时段。
8. 设置最少间隔或次数上限。
9. 查看当前判断结果。
10. 保存。

### 11.3 新建保护限制

1. 点击新建计划。
2. 选择保护限制。
3. 选择限制对象：全部自动动作、加药动作、排水动作、某个输出设备。
4. 选择限制条件：固定时段、维护状态、设备状态、传感器状态。
5. 设置生效星期和时段。
6. 保存。

### 11.4 暂停自动计划

暂停自动计划只影响后台自动执行，不影响手动点动和执行一次。

用户看到的文案：

- 自动计划已暂停。
- 手动执行仍可使用。
- 设备不会按计划自动启动。

## 12. 术语表

普通界面使用：

| 用户看到 | 含义 |
| --- | --- |
| 任务计划 | 自动执行安排 |
| 按时间执行 | 到时间后执行动作 |
| 按水质执行 | 水质条件满足后执行动作 |
| 保护限制 | 满足限制时禁止某类自动动作 |
| 生效时间 | 这条计划允许工作的星期和时段 |
| 执行动作 | 要调用的动作配置 |
| 检查配置 | 不动设备，只检查设置 |
| 执行一次 | 立即真实执行一次 |
| 暂停自动计划 | 停止后台自动执行 |
| 允许自动控制设备 | 允许计划真实控制继电器或 PWM |

普通界面禁止出现：

| 后端词 | 替代方式 |
| --- | --- |
| dry-run | 检查配置，或仅记录 |
| thread | 调度服务，仅放高级诊断 |
| task / rule / schedule | 计划 |
| aggregation | 判断方式 |
| fresh data window | 数据有效时间 |
| cooldown | 两次执行最少间隔 |
| hardware armed | 允许自动控制设备 |

## 13. 树莓派边缘端约束

### 13.1 屏幕形态

任务计划模块只按两种前端形态设计：

1. 10 寸机置屏幕：主操作界面，用于日常配置、调试和运维。
2. 手机窄屏：应急查看和少量操作，用于临时联网访问。

不再单独适配 7 寸主屏。小于 10 寸的屏幕可以落入手机/窄屏应急模式，但不作为完整编辑体验目标。

10 寸机置屏幕优先保证：

- 计划列表和当前计划详情能同时看到。
- 保存、检查配置、执行一次能在当前视野内完成。
- 当前计划为什么执行或没执行可以直接看到。
- 时间设置不依赖复杂图形控件。

手机应急模式优先保证：

- 查看自动计划是否运行。
- 查看某条计划最近结果和未执行原因。
- 暂停或恢复单条计划。
- 执行一次或检查配置。

手机应急模式不追求完整高级编辑，不承载复杂时间表达式配置。

### 13.2 前后端约束

前端要求：

- 不引入重型图表库来画计划编辑器。
- 不使用复杂动画。
- 不使用 canvas 作为主要交互。
- 使用原生表单、CSS grid、少量 JS 完成。
- 计划预览可以由前端简单计算，也可以由后端返回。

后端要求：

- 调度判断必须在本地完成。
- 配置保存在本地文件或本地数据库。
- 日志保存在本地。
- 断网不影响自动计划执行。
- 前端关闭不影响自动计划执行。

## 14. MVP 范围

第一版应该实现：

- 统一计划列表。
- 按时间执行。
- 按水质执行。
- 生效星期和生效时段。
- 执行动作选择。
- 启用和暂停单条计划。
- 检查配置。
- 执行一次。
- 运行记录。
- 普通文案去后端化。
- 高级诊断折叠。

第一版可以暂缓：

- 节假日模板。
- 可复用日历模板。
- 多设备复杂互锁。
- 复杂 PID 控制。
- 图形化拖拽编排。
- 云端同步。
- 用户权限体系。

## 15. 验收标准

- 用户无需理解后端 task、rule、schedule，也能新建一条时间计划。
- 用户无需理解后端 rule，也能新建一条 pH 条件加药计划。
- 用户能看懂某条计划为什么没有执行。
- 用户能区分“检查配置”和“执行一次”。
- 暂停自动计划不会阻断手动执行。
- 保护限制不会伪装成普通执行任务。
- 树莓派本地无公网时，已保存计划仍可自动运行。
- 横屏 1280x800 下不出现文字溢出和多层滚动混乱。
- 10 寸机置屏下可以完成计划查看、编辑、保存、检查配置、执行一次。
- 手机窄屏下可以完成查看、暂停、执行一次等应急动作。
- 普通界面不出现 dry-run、thread、aggregation、fresh window 等词。

## 16. 当前实现映射

现有后端仍可保留：

- action_task
- action_rule
- action_schedule
- action_unit
- automation runtime

前端任务计划模块需要做产品适配：

- action_schedule 在普通 UI 中表现为“按时间执行”。
- action_rule 在普通 UI 中表现为“按水质执行”。
- action_task 在普通 UI 中不作为一级对象展示，只作为执行动作的底层载体。
- automation runtime 在普通 UI 中表现为“自动计划是否运行”和“是否允许自动控制设备”。
- dry_run 不作为默认测试概念展示。

## 17. 最终判断

任务计划模块的核心不是增加规则数量，而是让现场用户形成稳定心智：

> 我知道系统什么时候会动，为什么会动，什么时候不会动，被谁拦住了，出了事去哪里查。

只要这个心智成立，后续再加高级时间表达式、保护限制模板、更多传感器条件，都不会把系统重新搞乱。
