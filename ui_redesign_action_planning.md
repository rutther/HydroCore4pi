# HydroCore 3.1 动作与计划中心重设计方案

## 1. 目标

这次重设计只解决一件事：

把现在分散在“动作配置”和“任务管理”里的概念，整理成用户能看懂、能配置、能放心执行的一套界面。

核心原则：

1. 前端只说用户语言，不说后端黑话。
2. 用户先想“我要设备什么时候做什么”，不是先想 `task / rule / schedule / thread`。
3. 单次动作、联动触发、营业时段、人工点动，全部落到同一套“计划”模型里。
4. 高风险输出必须有明显的安全状态和手动确认路径。
5. 横屏是主工作台，竖屏是巡检/修改模式，不能只是简单换行。

## 2. 新的信息架构

建议把现有顶部导航中的“动作配置”和“任务管理”重组为：

- 数据仪表
- 设备与采集
- 执行计划
- 系统设置

其中“执行计划”下的子页面固定为四个：

1. 输出设备
2. 动作模板
3. 计划中心
4. 运行记录

### 2.1 输出设备

回答“我到底控制哪几个口”。

每个输出设备卡片只展示：

- 名称
- 设备类型：继电器 / PWM
- GPIO 引脚
- 当前安全状态：锁定 / 可输出
- 手动测试按钮

用户不需要先理解动作、任务、规则，先把硬件对象认清楚。

### 2.2 动作模板

回答“这个口怎么动一次”。

动作模板是可复用的执行片段，面向设备动作本身，不讨论何时触发。

建议支持三类模板：

1. 单次动作  
   例如：加药泵 A 运行 3 秒
2. 循环动作  
   例如：运行 10 分钟，每分钟前 30 秒导通
3. PWM 动作  
   例如：PWM 通道 1 以 40% 输出 20 秒

这里保留你原来想要的“自由度”，但约束在动作层，不把触发逻辑混进去。

### 2.3 计划中心

回答“什么时候执行哪个动作”。

这是新的核心页面，用一个统一的“计划”对象替代原先分裂的：

- 任务
- 触发规则
- 定时计划

新的计划对象由三部分组成：

1. 执行内容  
   选择一个或多个动作模板
2. 触发方式  
   时间触发 / 条件触发 / 时间窗口 + 条件
3. 安全限制  
   最短间隔、每日上限、仅模拟执行、故障时禁止执行

也就是说：

- “每天 12:00 排水 20 秒”是计划
- “PH 连续 10 分钟高于阈值后执行加药”也是计划
- “周一到周六 06:00-20:00 期间，每小时执行一次循环泵动作”还是计划

触发只是计划的参数，不应该是单独的大栏目。

### 2.4 运行记录

回答“系统刚才做了什么，为什么做，为什么没做”。

每一条记录统一展示：

- 时间
- 计划名称
- 执行动作
- 来源：手动 / 时间触发 / 条件触发
- 结果：成功 / 模拟执行 / 跳过 / 阻止
- 原因说明

这比现在散在动作页、任务页里的各种日志更容易追。

## 3. 用户术语替换

下面这些词建议前端禁用：

- dry-run
- thread
- automation
- fresh data window
- cooldown
- aggregation
- single_action task
- run_action_unit

统一替换为：

- dry-run -> 仅模拟执行
- allow real output -> 允许真实输出
- thread running -> 调度服务运行中
- fresh data window -> 数据有效期
- cooldown -> 最短间隔
- aggregation -> 统计方式
- task -> 计划
- action unit -> 动作模板

## 4. 页面结构建议

## 4.1 横屏主工作台

适配场景：

- 1280x800
- 1366x768
- 1920x1080
- 长宽比 >= 1.25

布局采用三栏：

1. 左栏：计划列表 / 设备列表
2. 中栏：当前计划编辑器
3. 右栏：触发摘要、执行链路、安全状态、最近记录

这样可以一边选，一边改，一边看结果，不需要频繁切页。

## 4.2 竖屏巡检模式

适配场景：

- 800x1280
- 1080x1920
- 长宽比 < 1.25

布局改成分步骤卡片：

1. 先选计划
2. 再看触发条件
3. 再看执行动作
4. 最后执行或保存

竖屏不能简单把三栏压成一列，否则会变成长到看不完的表单。

## 5. 计划中心的数据模型建议

前端产品层建议统一为：

```json
{
  "id": "plan_dose_a_workhours",
  "name": "工作时段加药 A",
  "enabled": true,
  "actions": ["dose_a_3s"],
  "trigger": {
    "mode": "time_and_condition",
    "time_window": {
      "days": ["mon", "tue", "wed", "thu", "fri", "sat"],
      "start": "06:00",
      "end": "20:00",
      "every_minutes": 60
    },
    "condition": {
      "metric": "ph_value",
      "operator": ">",
      "threshold": 7.8,
      "sustain_sec": 600,
      "stat": "avg",
      "data_ttl_sec": 90
    }
  },
  "limits": {
    "min_interval_sec": 1800,
    "max_runs_per_day": 12
  },
  "runtime": {
    "simulation_only": true
  }
}
```

后端内部仍然可以保留现有 `action_unit / task / rule / schedule` 结构做兼容，但前端不应该再直接暴露这几层。

## 6. 动作模板和计划的边界

建议边界这样定：

### 动作模板负责

- 控制哪个输出
- 运行多久
- 连续还是脉冲
- PWM 占空比多少
- 是否允许手动测试

### 计划负责

- 什么时候执行
- 满足什么条件时执行
- 能不能重复执行
- 执行频率限制
- 在什么时间段内有效

这样就不会出现“动作里写了一半触发逻辑，计划里又写一半”的冲突。

## 7. 这次预览稿的视觉方向

视觉上不再走“霓虹赛博控制台”路线，改成更像工业设备 HMI：

- 深色底，但不是纯黑
- 面板边界明确
- 重点信息用青绿和琥珀色
- 危险操作用红色
- 按钮形状统一为 8px 圆角
- 标签、状态、字段说明尽量短

目标是：

看上去像能稳定运行的设备系统，而不是开发中的实验页面。
