# HydroCore 3.1 动作层重设计方案

## 1. 这层到底要解决什么

动作层不是“GPIO 配置器”，而是把采集系统变成控制系统的那一层。

它要解决 4 件事：

1. 定义这台树莓派上哪些输出口真的接了什么设备。
2. 把底层 GPIO 动作包装成“人能理解的动作”。
3. 让这些动作可以被手动执行，也可以被规则自动触发。
4. 把每一次动作都记录下来，方便追责、排障、回看。

如果只暴露 GPIO、时长、占空比，系统会很快退化成工程调试台，不适合长期使用。

所以产品上应该把控制相关能力拆成：

- 输出设备
- 动作模板
- 任务管理
- 自动规则
- 执行记录

这 5 层刚好对应“接了什么”“怎么动”“谁来调度”“什么时候动”“动过什么”。

## 2. 站在不同角色眼里，这东西应该怎么被理解

### 2.1 安装者

他关心的是：

- 这根线接到了哪个 GPIO
- 这个继电器是高电平触发还是低电平触发
- 上电默认应该是开还是关
- 能不能点一下测试

所以安装者面对的对象应该叫“输出设备”，而不是“动作单元”。

### 2.2 运营者

他关心的是：

- 我要手动开一下排水阀
- 我要让 1 号加药泵跑 3 秒
- 这个动作有没有真的执行

所以运营者面对的对象应该叫“动作”，而不是 GPIO。

### 2.3 维护者

他关心的是：

- 什么时候触发过
- 为什么触发
- 失败在哪一步
- 是手动执行还是自动规则执行

所以维护者需要“执行记录”和“规则命中原因”。

## 3. 这套硬件的建议对象模型

已知硬件目标：

- 3 个开关继电器 GPIO
- 1 个 PWM GPIO

建议先抽象成 4 个输出设备：

1. `dose_pump_a`
   - 名称：加药泵 A
   - 类型：relay
   - 用途：某种药剂泵，通电即运行

2. `dose_pump_b`
   - 名称：加药泵 B
   - 类型：relay
   - 用途：另一种药剂泵，通电即运行

3. `drain_valve`
   - 名称：排水阀
   - 类型：relay
   - 用途：排水/换水

4. `aux_pwm_1`
   - 名称：PWM 通道 1
   - 类型：pwm
   - 用途：先定义为通用 PWM，未来可挂调速泵、风扇、电磁比例阀等

注意：PWM 口现在不要假装已经知道用途。产品里就叫“PWM 通道 1”，保留空间更稳。

## 4. 产品模型：从 3 层改成 5 层

旧思路容易变成：

- GPIO
- 动作
- 计划

这太薄，后面规则一复杂就会缠在一起。

建议改成下面 5 层：

### 4.1 输出设备

定义真实硬件接线。

字段建议：

- `id`
- `name`
- `kind`: `relay` | `pwm`
- `gpio_pin`
- `active_level`: `high` | `low`，仅 relay
- `safe_state`: `off` | `on`
- `pwm_frequency`
- `default_duty`
- `enabled`
- `allow_real_output`
- `description`

例子：

```json
{
  "id": "dose_pump_a",
  "name": "加药泵 A",
  "kind": "relay",
  "gpio_pin": 17,
  "active_level": "low",
  "safe_state": "off",
  "enabled": true,
  "allow_real_output": false
}
```

### 4.2 动作模板

把底层 IO 包成可复用动作。

动作模板应该是“业务动作”，不是“GPIO 指令集”。

建议先支持 3 类：

1. `pulse`
   - 打开某输出，持续 N 秒，再关闭
   - 适合加药泵、阀门点动

2. `hold`
   - 切换到某状态，保持不自动回退
   - 适合手动开启排水阀、手动停泵

3. `pwm_run`
   - 以某占空比运行一段时间
   - 适合未来 PWM 输出

字段建议：

- `id`
- `name`
- `target_output_id`
- `action_type`
- `params`
- `cooldown_sec`
- `confirm_required`
- `enabled`
- `description`

例子：

```json
{
  "id": "dose_a_3s",
  "name": "加药泵 A 运行 3 秒",
  "target_output_id": "dose_pump_a",
  "action_type": "pulse",
  "params": {
    "duration_ms": 3000
  },
  "cooldown_sec": 30,
  "confirm_required": false,
  "enabled": true
}
```

这层非常重要，因为用户以后点的是“加药泵 A 运行 3 秒”，不是“GPIO17 拉低 3000ms”。

### 4.3 任务管理

这是这次补上的关键层。

任务管理不直接控制 GPIO，它负责“调用动作模板”。

这样做的好处是：

- 手动执行、规则触发、定时执行都走同一条执行链路
- 日志模型统一
- 后续插入暂停、禁用、审计、限流会简单很多

任务应该理解为“一个可被调度的控制任务”。

第一版建议只支持 2 类任务：

1. `single_action`
   - 调一次动作模板
   - 例子：执行“加药泵 A 运行 3 秒”

2. `sequence`
   - 顺序调多个动作模板，中间可等待
   - 例子：打开排水阀 -> 等待 20 秒 -> 关闭排水阀

注意：这里的 sequence 是任务编排，不是把底层 GPIO 暴露给用户。

字段建议：

- `id`
- `name`
- `task_type`
- `steps`
- `enabled`
- `cooldown_sec`
- `confirm_required`
- `description`

例子：

```json
{
  "id": "drain_20s",
  "name": "排水 20 秒",
  "task_type": "sequence",
  "enabled": true,
  "cooldown_sec": 60,
  "steps": [
    {
      "step_type": "run_action",
      "action_template_id": "drain_valve_open"
    },
    {
      "step_type": "wait",
      "duration_ms": 20000
    },
    {
      "step_type": "run_action",
      "action_template_id": "drain_valve_close"
    }
  ]
}
```

对用户来说，“任务”比“动作模板”更像一个真正可执行的操作对象。

动作模板是原子能力，任务是被调用单元。

### 4.4 自动规则

这层决定“什么时候自动执行哪个动作”。

规则不要做成脚本，也不要做成通用编排引擎。这个产品不是低代码平台。

规则也不要直接调 GPIO，应该调用任务。

建议第一版只做条件规则：

- 当某指标超过阈值
- 持续达到多久
- 执行某任务
- 然后冷却多久

字段建议：

- `id`
- `name`
- `enabled`
- `metric_key`
- `operator`: `>` `<` `>=` `<=`
- `threshold`
- `sustain_sec`
- `task_id`
- `cooldown_sec`
- `max_runs_per_hour`
- `requires_fresh_data`
- `description`

例子：

```json
{
  "id": "ph_high_dose_a",
  "name": "pH 偏高时启动加药泵 A",
  "enabled": false,
  "metric_key": "ph",
  "operator": ">",
  "threshold": 6.8,
  "sustain_sec": 30,
  "task_id": "task_dose_a_3s",
  "cooldown_sec": 600,
  "max_runs_per_hour": 4,
  "requires_fresh_data": true
}
```

### 4.5 定时计划

除了“数据触发规则”，还要有“时间触发计划”。

这部分不应该混在自动规则里，也不应该复用采集任务计划。

因为它回答的问题不同：

- 自动规则：什么时候因为数据变化而执行
- 定时计划：什么时候因为时间到了而执行

所以建议把“任务管理”再拆成两种调用来源：

1. 规则调用
2. 定时调用

定时计划第一版建议只支持 3 类：

1. `once`
   - 某个具体时间执行一次

2. `daily`
   - 每天某个时间执行

3. `interval`
   - 每隔 N 分钟执行一次

字段建议：

- `id`
- `name`
- `enabled`
- `schedule_type`
- `task_id`
- `time_of_day`
- `interval_sec`
- `start_at`
- `end_at`
- `cooldown_sec`
- `skip_if_task_running`
- `description`

例子：

```json
{
  "id": "daily_drain_noon",
  "name": "每天中午排水 20 秒",
  "enabled": false,
  "schedule_type": "daily",
  "task_id": "drain_20s",
  "time_of_day": "12:00",
  "cooldown_sec": 600,
  "skip_if_task_running": true
}
```

### 4.6 执行记录

必须可追踪，且不是附属功能。

每条记录至少要包含：

- 时间
- 来源：`manual` / `rule`
- 规则名或动作名
- 目标输出
- dry-run 还是真执行
- 成功/失败/阻断
- 原因
- 关键参数快照

## 5. 面向用户的操作路径

动作层不应该一上来就扔给用户四个列表。

正确的路径应该是：

### 第一步：配置输出设备

用户完成：

- 选择 GPIO
- 填设备名
- 指定 relay / pwm
- 指定触发电平
- 点一次测试

完成后系统知道“硬件线是怎么接的”。

### 第二步：定义动作模板

用户完成：

- 选目标设备
- 选动作类型
- 填持续时间 / 占空比 / 目标状态
- 命名

完成后系统知道“要怎么动”。

### 第三步：手动测试动作模板

用户从动作列表里点执行：

- 默认 dry-run
- 开启“真实输出”后才允许真执行
- 真执行前要弹确认

完成后系统知道“这动作在硬件上跑不跑得通”。

### 第四步：定义任务

用户完成：

- 直接选一个动作模板作为任务
- 或把多个动作模板串成一个顺序任务
- 设置冷却时间

完成后系统知道“后续到底执行什么”。

### 第五步：配置自动规则

用户完成：

- 选指标
- 选阈值和持续时间
- 选要触发的任务
- 设冷却时间和频次限制

完成后系统知道“什么时候自动动”。

### 第六步：配置定时计划

用户完成：

- 选要执行的任务
- 选执行方式：一次 / 每天 / 间隔
- 设时间或周期
- 决定冲突时跳过还是等待

完成后系统知道“什么时候按时间动”。

### 第七步：看记录

用户查看：

- 最近执行过哪些动作
- 为什么执行
- 为什么失败
- 是否存在频繁触发

## 6. 前端信息架构

动作层建议单独做成 6 个标签页，而不是混在一堆工程页面里。

### 6.1 总览

显示：

- 4 个输出设备的当前状态
- 最近 10 条执行记录
- 已启用自动规则数量
- 是否处于 dry-run 模式
- 是否检测到真实 GPIO 驱动

这是用户进入动作层的第一屏。

### 6.2 输出设备

每张卡代表一个真实输出：

- 名称
- GPIO
- 类型
- 触发方式
- 是否启用
- 是否允许真实输出
- 测试按钮

这是安装者页面。

### 6.3 动作模板

列表显示：

- 动作名
- 目标设备
- 动作类型
- 关键参数
- 冷却时间
- 执行按钮

编辑器不要让用户直接写 JSON。

用表单分情况渲染：

- `pulse`: 时长输入框
- `hold`: 开/关选择
- `pwm_run`: 占空比 + 时长

### 6.4 任务管理

这里列出的应该是“可被调用的任务”。

列表显示：

- 任务名
- 任务类型
- 组成内容
- 冷却时间
- 手动执行按钮

编辑器分两种：

- 单动作任务：选择一个动作模板
- 顺序任务：动作模板 + 等待 + 动作模板

### 6.5 自动规则

规则编辑器做成一句话式：

`当 [pH] [>] [6.8] 且持续 [30 秒] 时，执行 [加药泵 A 校正任务]，冷却 [10 分钟]`

这是最符合人类理解的方式。

### 6.6 定时计划

定时计划编辑器也做成一句话式：

- `每天 [12:00] 执行 [排水 20 秒]`
- `每隔 [30 分钟] 执行 [加药泵 B 运行 1 秒]`

重点是让用户觉得自己在设“计划”，不是在写 cron。

### 6.7 执行记录

按时间倒序：

- 时间
- 来源
- 动作
- 输出设备
- 结果
- 消息
- 展开详情

失败记录用明显颜色标出。

## 7. 为什么不建议继续用“动作单元 steps JSON”

原因不是技术不行，而是产品收益不高。

对这个项目来说：

- 输出设备数量很少
- 动作类型很稳定
- 用户不是程序员
- 规则本身才是后期复杂度来源

所以让用户编辑步骤 JSON 属于暴露内部实现，收益小，认知负担大。

如果将来真的需要复杂编排，可以保留一个“高级模式”，但不应该是默认入口。

## 8. 为什么要把“动作模板”和“任务管理”分开

因为这两层解决的不是同一个问题。

- 动作模板：定义单个设备怎么动
- 任务管理：定义一个完整操作怎么被调用

举例：

- “加药泵 A 运行 3 秒”是动作模板
- “pH 校正一次”可以是任务
- “排水 20 秒”也可以是任务

这样规则和定时计划都只调用任务，不需要理解底层动作细节。

对用户来说更简单，对系统来说也更稳定。

## 9. 约束和安全策略

这部分必须做严，不然继电器控制迟早出事故。

### 8.1 默认 dry-run

系统初始只允许 dry-run。

只有同时满足以下条件才允许真输出：

- 检测到真实 GPIO 驱动
- 输出设备 `allow_real_output = true`
- 用户在界面明确开启真输出

### 8.2 安全上电

服务启动时：

- 初始化所有已启用输出
- 立即恢复到 `safe_state`
- 写系统日志

### 8.3 单输出互斥锁

任何时刻，同一个输出设备只能有一个动作占用。

例如：

- “排水阀打开”
- “排水阀关闭”

不能并发。

### 9.4 规则冷却

自动规则一定要有：

- 持续时间
- 冷却时间
- 每小时最大执行次数

否则阈值抖动会疯狂触发。

### 9.5 数据新鲜度

规则判断时要检查采样是否新鲜。

比如最近 2 分钟没有新数据，就不要继续自动加药。

### 9.6 任务并发策略

任务层还要定义一个简单并发策略：

- 同名任务是否允许重复进入
- 任务运行时，新的同名触发是跳过还是排队

第一版建议：

- 默认不重入
- 默认跳过并记日志

### 9.7 人工总闸

建议做一个全局总开关：

- 自动规则启用 / 禁用
- 定时计划启用 / 禁用

这样现场出问题时，用户能一键冻结自动控制，但保留手动动作。

## 10. 与现有系统的边界关系

### 9.1 采集计划不负责动作

`config_poll_plan.json` 继续只管采集。

不要把动作规则塞进去。

### 10.2 动作层独立存储

建议独立目录：

- `data/actuators/`
- `data/action_templates/`
- `data/action_tasks/`
- `data/action_rules/`
- `data/action_schedules/`

### 10.3 执行日志进 SQLite

这部分已经走对了，继续用库表。

建议后续扩展字段：

- `trigger_rule_id`
- `schedule_id`
- `task_id`
- `output_id`
- `dry_run`
- `duration_ms`
- `result_code`
- `detail_json`

## 11. 推荐的第一版真实业务命名

为了让前端像产品，不像脚手架，建议默认创建这 4 个输出和 6 个动作模板。

### 输出设备

1. 加药泵 A
2. 加药泵 B
3. 排水阀
4. PWM 通道 1

### 动作模板

1. 加药泵 A 运行 1 秒
2. 加药泵 A 运行 3 秒
3. 加药泵 B 运行 1 秒
4. 加药泵 B 运行 3 秒
5. 排水阀打开
6. 排水阀关闭

PWM 动作先给 2 个模板：

7. PWM 通道 1 运行 30% / 10 秒
8. PWM 通道 1 运行 60% / 10 秒

这样一进入系统就有“真实世界对象”，不是空壳。

### 任务

1. pH 校正一次（调用 加药泵 A 运行 3 秒）
2. EC 校正一次（调用 加药泵 B 运行 3 秒）
3. 排水 20 秒（打开排水阀 -> 等待 -> 关闭）

### 定时计划

1. 每天中午排水 20 秒
2. 每 6 小时执行一次 PWM 通道 1 低速运行

## 12. 规则第一版建议只做这些能力

做少一点，反而更稳。

建议只支持：

1. 单指标阈值
2. 持续时间
3. 冷却时间
4. 频次限制
5. 启用 / 禁用

先不要支持：

- 多条件 AND/OR
- 脚本表达式
- 多步流程编排
- 条件嵌套
- 时间窗调度

这些可以以后再加，但第一版不值得。

## 13. 任务管理第一版建议只做这些能力

建议只支持：

1. 单动作任务
2. 顺序任务
3. 一次性计划
4. 每日计划
5. 间隔计划

先不要支持：

- cron 表达式
- 周历排程
- 条件分支
- 并行步骤
- 失败补偿流程

这样用户才不会被复杂度淹没。

## 14. 开发落地建议

建议按下面顺序推进，而不是一次性全做：

### Phase 1：输出设备

- 完成 4 个输出设备模型
- 完成 GPIO 配置和测试
- 完成真实输出保护

### Phase 2：动作模板

- 去掉 JSON steps 编辑
- 改成表单式动作编辑
- 完成手动执行闭环

### Phase 3：任务管理

- 完成任务模型
- 完成单动作任务和顺序任务
- 统一所有执行入口

### Phase 4：自动规则

- 完成规则存储
- 完成规则轮询判定
- 完成冷却/频控/数据新鲜度保护

### Phase 5：定时计划

- 完成 once / daily / interval
- 完成调度器
- 完成冲突跳过与日志

### Phase 6：总览与维护

- 做动作总览页
- 做告警和失败高亮
- 做日志筛选

## 15. 最终建议

对这个项目来说，动作层最好的产品形态不是“万能自动化编排器”，而是“极少配置、明确对象、足够安全的本地控制台”。

所以我建议把它收敛成一句话：

> 先定义 4 个输出设备，再定义几个能直接看懂的动作模板，把这些模板包装成任务，然后用规则或定时计划去调用任务。

这比“模块调模块”的工程视角更接近用户，也更适合你这个项目现在的体量和硬件规模。
